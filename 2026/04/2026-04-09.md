## Ask Copilot in security assessments now available

**URL:** https://github.blog/changelog/2026-04-09-ask-copilot-in-security-assessments-now-available

### 要約
セキュリティ評価の結果画面から、Copilot に直接コンテキスト付きの説明と次の手順を聞けるようになりました。対象は secret risk assessment と Code Security risk assessment の結果です。

### ユーザーにとって良いこと
評価結果の読み解きにかかる時間を減らし、修正の優先順位付けを早められます。管理者やセキュリティ担当が、その場で実行可能な次の一手を把握しやすくなります。

### アクションプラン
- Organization のセキュリティ担当者に、評価結果画面から Copilot を使えることを周知してください。
- secret risk assessment / Code Security risk assessment の運用手順に、Copilot への確認を追加してください。
- この機能は結果画面からの利用が前提なので、対象権限を持つ admin / security manager に限定して案内してください。

### リファレンス
- [Ask Copilot in security assessments now available](https://github.blog/changelog/2026-04-09-ask-copilot-in-security-assessments-now-available)

## New Low Quality option in the Hide comment menu

**URL:** https://github.blog/changelog/2026-04-09-new-low-quality-option-in-the-hide-comment-menu

### 要約
コメントの非表示時に、新しい分類として `Low Quality` を選べるようになりました。Issues、Discussions、Pull Request、Commit のコメントで利用できます。

### ユーザーにとって良いこと
スパムや abuse ほどではない、低品質コメントをより適切に分類できます。モデレーションの記録精度が上がり、コミュニティ運営の分析もしやすくなります。

### アクションプラン
- モデレーション担当に `Low Quality` の用途を案内してください。
- 既存の運用ルールで、Spam / Abuse / Low Quality の分類基準を整理してください。
- この分類はモデレーション向けのため、一般ユーザー向けの手順と混同しないようにしてください。

### リファレンス
- [New Low Quality option in the Hide comment menu](https://github.blog/changelog/2026-04-09-new-low-quality-option-in-the-hide-comment-menu)

## New Sort by control added to Notifications

**URL:** https://github.blog/changelog/2026-04-09-new-sort-by-control-added-to-notifications

### 要約
Notifications ページに並び順を切り替える `Sort by` が追加されました。`Newest to oldest` と `Oldest to newest` を選べ、選択はセッションをまたいで保持されます。

### ユーザーにとって良いこと
古い通知から順に処理できるため、たまりがちな通知を計画的に消化しやすくなります。リポジトリ単位でグループ化している場合も、グループ順と項目順の両方に反映されます。

### アクションプラン
- 通知を定期処理しているチームは、`Oldest to newest` を使う運用を検討してください。
- サポート担当や triage 担当には、通知の処理順を切り替えられることを共有してください。
- 既存の通知ワークフローに変更は不要ですが、並び順の既定値が `Newest to oldest` のままである点を把握してください。

### リファレンス
- [New Sort by control added to Notifications](https://github.blog/changelog/2026-04-09-new-sort-by-control-added-to-notifications)

## Repository member role labels now in pull request list view

**URL:** https://github.blog/changelog/2026-04-09-repository-member-role-labels-now-in-pull-request-list-view

### 要約
公開リポジトリの pull request 一覧で、Contributor や Member などのロールラベルが直接表示されるようになりました。個別の PR を開かなくても、投稿者の文脈を一覧で把握できます。

### ユーザーにとって良いこと
初回投稿者やメンバーをすぐ判別できるため、triage が速くなります。新規貢献者を見落としにくくなり、歓迎やレビューの優先付けもしやすくなります。

### アクションプラン
- PR triage の際は、一覧表示のロールラベルを確認する前提で運用してください。
- First-time contributor を優先的に扱うガイドラインがある場合は、一覧画面ベースで更新してください。
- `Member` ラベルの可視性は設定により変わるため、公開範囲の前提をチーム内で揃えてください。

### リファレンス
- [Repository member role labels now in pull request list view](https://github.blog/changelog/2026-04-09-repository-member-role-labels-now-in-pull-request-list-view)

## Release information in issue sidebar and default values for project fields

**URL:** https://github.blog/changelog/2026-04-09-release-info-in-issue-sidebar-and-project-defaults

### 要約
Issue の sidebar で、リンクされた PR が最初に含まれた release 情報を確認できるようになりました。あわせて、Project の text / number / single select フィールドに default 値を設定でき、関連 issue への移動もやりやすくなりました。

### ユーザーにとって良いこと
修正がいつ出荷されたかを Issue 画面からすぐ追えるため、問い合わせ対応や確認作業が楽になります。Project の新規アイテム作成時も、初期値を自動適用できて入力漏れを減らせます。

### アクションプラン
- Issue 運用で release 追跡が必要な場合は、sidebar の Development セクションを確認する手順を追加してください。
- Project の新規アイテムに毎回入れる値があるなら、default values を設定してください。
- Issue viewer で開く挙動が追加されたので、コメントやタイムラインからの参照確認フローを見直してください。

### リファレンス
- [Release information in issue sidebar and default values for project fields](https://github.blog/changelog/2026-04-09-release-info-in-issue-sidebar-and-project-defaults)
