## OIDC support for Dependabot and code scanning

**URL:** https://github.blog/changelog/2026-04-14-oidc-support-for-dependabot-and-code-scanning

### 要約
Dependabot と code scanning が、organization レベルで OIDC を使って private registry にアクセスできるようになりました。長期保存の秘密情報を repository secret に置かず、クラウド IdP から短期資格情報を動的に取得できます。既存の repo-level 設定を organization-wide に拡張した形です。

### ユーザーにとって良いこと
registry 認証の運用負荷と secret 管理リスクを下げられます。複数リポジトリで一貫したアクセス制御を中央管理しやすくなります。

### アクションプラン
- Organization admin は Dependabot / code scanning 用の OIDC registry 設定を有効化してください。
- `AWS CodeArtifact`、`Azure DevOps Artifacts`、`JFrog Artifactory` の利用可否を確認し、移行対象を洗い出してください。
- ⚠️ Cloudsmith と Google Artifact Registry は今後 4 週間で追加予定のため、現時点では未対応です。

## Deployment context in repository properties and alerts

**URL:** https://github.blog/changelog/2026-04-14-deployment-context-in-repository-properties-and-alerts

### 要約
repository properties に `deployable` と `deployed` が追加され、デプロイ状態を metadata から自動判定できるようになりました。あわせて、Dependabot と code scanning の alert 画面に runtime risk context が表示されます。どちらも generally available です。

### ユーザーにとって良いこと
デプロイ済みのリポジトリにだけ厳しいポリシーを当てるなど、運用とセキュリティ制御を実態に合わせやすくなります。alert の優先順位付けも、実行時コンテキストを見て素早くできます。

### アクションプラン
- Repository properties を使って、rulesets や branch protection の対象を deployment 状態で絞り込んでください。
- Security team は alert triage の基準に runtime context を追加してください。
- 既存の手動ラベルや一覧管理を、`deployable` / `deployed` ベースに置き換えられないか確認してください。

### リファレンス
- [Deployment context in repository properties and alerts](https://github.blog/changelog/2026-04-14-deployment-context-in-repository-properties-and-alerts)

## Link code scanning alerts to GitHub Issues

**URL:** https://github.blog/changelog/2026-04-14-link-code-scanning-alerts-to-github-issues

### 要約
code scanning alerts を GitHub Issues に直接リンクできるようになりました。alert 側の Tracking セクションや issue 側の Relationships から相互に関連付けできます。alert 一覧では tracking 状態のアイコンと `has:tracking` / `no:tracking` フィルタも使えます。

### ユーザーにとって良いこと
脆弱性の修正計画を issue ベースで管理しやすくなります。未対応 alert を見つけやすく、既存の作業トラッキングに security remediation を組み込みやすくなります。

### アクションプラン
- code scanning を使っているリポジトリで、既存 issue 運用に alert tracking を組み込んでください。
- `has:tracking` と `no:tracking` を使って、未追跡 alert の棚卸しを実施してください。
- ⚠️ public preview のため、運用フローに入れる前に一部リポジトリで試験導入してください。

### リファレンス
- [Link code scanning alerts to GitHub Issues](https://github.blog/changelog/2026-04-14-link-code-scanning-alerts-to-github-issues)

## Secret scanning pattern updates and product improvements

**URL:** https://github.blog/changelog/2026-04-14-secret-scanning-pattern-updates-and-product-improvements

### 要約
Secret scanning に Cloudflare detectors が追加され、push protection の既定範囲や API 機能も拡張されました。EMUs の fork 継承、campaign の team/topic フィルタ、alert API の provider フィールド、scan history の generic secrets backfill などが含まれます。Enterprise 向けには dismissal request を一覧する API も追加されました。

### ユーザーにとって良いこと
検出範囲が広がり、誤って秘密情報を push するリスクを減らせます。API と campaign 管理の改善で、大規模な secret remediation を自動化しやすくなります。

### アクションプラン
- Cloudflare 関連 secret の検出対象を security policy に反映してください。
- campaign 運用で team/topic フィルタを使い、担当範囲ごとに絞り込んでください。
- alert API を利用する自動化は `provider` / `provider_slug` に対応させてください。
- ⚠️ `providers` と `exclude_providers` は同時指定できないため、既存クエリを見直してください。

## SBOM exports are now computed asynchronously

**URL:** https://github.blog/changelog/2026-04-14-sbom-exports-are-now-computed-asynchronously

### 要約
Dependency graph の SBOM export が非同期処理になり、UI と API が新しい job-based の流れに変わりました。大きいリポジトリでも timeout しにくくなり、重複リクエストが別 worker を無駄に起動する問題も避けられます。新しい generate/fetch の 2 つの API が提供されます。

### ユーザーにとって良いこと
大規模リポジトリでも SBOM を安定して取得しやすくなります。待ち時間の制御が改善され、ブラウザと API の両方で扱いやすくなります。

### アクションプラン
- 既存の SBOM 取得スクリプトを新しい `generate-report` / `fetch-report` フローに更新してください。
- UI で dependency graph から SBOM を取得する手順を、利用者向けに案内してください。
- ⚠️ SBOM は HEAD のみ対象で、匿名ユーザーはリポジトリごとに同時 1 件までに制限されます。

### リファレンス
- [SBOM exports are now computed asynchronously](https://github.blog/changelog/2026-04-14-sbom-exports-are-now-computed-asynchronously)

## Model selection for Claude and Codex agents on github.com

**URL:** https://github.blog/changelog/2026-04-14-model-selection-for-claude-and-codex-agents-on-github-com

### 要約
github.com 上の Claude と Codex の third-party coding agents で、タスク開始時に model selection ができるようになりました。Claude では Sonnet 4.6 / Opus 4.6 / Sonnet 4.5 / Opus 4.5、Codex では GPT-5.2-Codex / GPT-5.3-Codex / GPT-5.4 が選べます。Copilot cloud agent と同じく、より適したモデルを選択できます。

### ユーザーにとって良いこと
作業内容に応じてモデルを選べるため、精度や応答特性を調整しやすくなります。チームごとの agent 運用を、より柔軟に設計できます。

### アクションプラン
- Claude / Codex agent の利用方針に、モデル選択ルールを追加してください。
- Enterprise 管理者は Anthropic Claude と OpenAI Codex の policy を有効化してください。
- リポジトリ所有者は Settings > Copilot > Cloud agent で agent を有効化してください。

## GitHub Code Quality: Improvements to standard findings in public preview

**URL:** https://github.blog/changelog/2026-04-14-github-code-quality-improvements-to-standard-findings-in-public-preview

### 要約
GitHub Code Quality の standard findings 画面が改善され、file path 検索、複数 findings の一括 dismiss / reopen、詳細な message と related locations の表示ができるようになりました。default branch 上の信頼性・保守性の問題を追いやすくする更新です。public preview と ECH data residency で利用できます。

### ユーザーにとって良いこと
大量の findings を効率よく整理でき、ノイズを減らしやすくなります。診断情報が見やすくなるため、修正判断も速くなります。

### アクションプラン
- public preview 参加中のリポジトリで、既存 findings の整理に検索と一括操作を使ってください。
- rule page の diagnostic message と related locations をレビュー手順に組み込んでください。
- ⚠️ preview 機能のため、本番運用の基準にする前に対象チームで検証してください。
