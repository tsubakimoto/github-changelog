## More visibility into Copilot coding agent sessions

**URL:** https://github.blog/changelog/2026-03-19-more-visibility-into-copilot-coding-agent-sessions

### 要約
Copilot coding agent のセッションログが強化され、リポジトリのクローンや agent firewall の起動といった組み込みセットアップ、`copilot-setup-steps.yml` の出力、サブエージェントの活動状況が見やすくなりました。これにより、Copilot が作業を始める前後や内部で何をしているかを追いやすくなります。

### ユーザーにとって良いこと
Copilot に作業を委譲した際の待ち時間や停止箇所の理由が分かりやすくなります。カスタムセットアップやサブエージェントの挙動も GitHub Actions の詳細ログまで潜らずに確認しやすくなり、調査と運用の負荷が下がります。

### アクションプラン
- Copilot coding agent を使っているリポジトリでは、次回の委譲タスクでセッションログに組み込みセットアップとサブエージェント表示が追加されていることを確認
- `copilot-setup-steps.yml` を運用している場合は、出力内容を見直して環境構築の失敗や不要なログ出力がないか点検
- 自動適用の改善として、まずは GitHub Actions の詳細ログを開く前にセッションログを確認する運用へ切り替え

### リファレンス
- [More visibility into Copilot coding agent sessions](https://github.blog/changelog/2026-03-19-more-visibility-into-copilot-coding-agent-sessions)

## GitHub Actions: Late March 2026 updates

**URL:** https://github.blog/changelog/2026-03-19-github-actions-late-march-2026-updates

### 要約
GitHub Actions に 2 つの改善が入りました。環境を secrets や variables の管理目的だけで使う場合に `deployment: false` を指定して deployment を作らず利用できるようになり、さらに scheduled workflows で IANA タイムゾーンを指定できるようになりました。

### ユーザーにとって良いこと
環境変数の管理だけを目的とした workflow で不要な deployment を増やさずに済みます。cron を UTC へ換算せずにローカル時刻で管理できるため、定期実行の保守性と可読性も向上します。

### アクションプラン
- `.github/workflows/*.yml` を見直し、環境の secrets / variables だけを使いたいジョブには `deployment: false` の採用を検討
- `on.schedule` を使う workflow では、UTC 前提の cron を `timezone: "Asia/Tokyo"` などの明示指定へ更新できるか確認
- ⚠️ custom deployment protection rule を使っている環境では `deployment: false` を利用できないため、対象の environment を先に棚卸し

### リファレンス
- [GitHub Actions: Late March 2026 updates](https://github.blog/changelog/2026-03-19-github-actions-late-march-2026-updates)

## Actions Runner Controller release 0.14.0

**URL:** https://github.blog/changelog/2026-03-19-actions-runner-controller-release-0-14-0

### 要約
GitHub Actions Runner Controller (ARC) 0.14.0 が一般提供になりました。runner scale set の複数ラベル対応、`actions/scaleset` クライアントへの移行、内部リソースへのラベル/注釈の付与、listener pod の Linux ノード既定化などが含まれます。

### ユーザーにとって良いこと
セルフホストランナーの設計自由度が上がり、OS、性能、ネットワーク、コンプライアンス要件ごとに scale set を細かく分けなくても運用しやすくなります。混在クラスタでの誤スケジューリング防止や、古い runner 構成での無駄なオートスケール抑止にも役立ちます。

### アクションプラン
- ARC を利用している場合は、staging 環境で 0.14.0 と Helm chart 差分を検証し、既存の `runs-on` 指定との整合性を確認
- runner scale set を用途別に細かく分けている場合は、複数ラベル対応で統合できる構成がないか見直し
- `listenerTemplate` や `nodeSelector` のカスタマイズをしている場合は、Linux ノード既定値との競合がないか確認
- Experimental Helm charts は任意なので、本番導入前に検証環境でのみ評価

### リファレンス
- [Actions Runner Controller release 0.14.0](https://github.blog/changelog/2026-03-19-actions-runner-controller-release-0-14-0)

## Codespaces with data residency now available in Japan

**URL:** https://github.blog/changelog/2026-03-19-codespaces-with-data-residency-now-available-in-japan

### 要約
Codespaces の data residency 対応リージョンに Japan が追加されました。GitHub Enterprise with data residency の public preview において、既存の EU と Australia に加えて日本リージョンでも Codespaces を利用できるようになります。

### ユーザーにとって良いこと
日本国内のデータ所在地要件がある組織でも、Codespaces の導入検討を進めやすくなります。クラウド開発環境の利便性を享受しつつ、リージョン要件や社内ガバナンスへの適合性を高められます。

### アクションプラン
- enterprise が Japan region でホストされているかを確認し、対象なら Codespaces の public preview 評価を開始
- データ所在地やセキュリティ審査の資料に Japan 対応を追記し、導入障壁を下げる
- オプトイン型の preview 機能として、利用候補チームを限定して接続性と運用性を先行確認

## View code and comments side-by-side in pull request Files changed page

**URL:** https://github.blog/changelog/2026-03-19-view-code-and-comments-side-by-side-in-pull-request-files-changed-page

### 要約
新しい pull request の `Files changed` ページで、Comments、Overview、Merge status、Alerts の docked panel が使えるようになりました。コード差分を見ながら、レビューコメント、PR 説明、マージ可否、セキュリティアラートを横並びで確認できます。

### ユーザーにとって良いこと
レビュー中の画面遷移が減り、文脈を失わずに PR の状態や会話を把握できます。コードレビュー、マージ準備、セキュリティ確認を同じ画面で回せるため、レビュー効率が上がります。

### アクションプラン
- まだ classic の `Files changed` ページを使っている場合は、新しい体験へ opt-in して docked panels を有効化
- レビュアー向けに `Comments`、`Overview`、`Merge status`、`Alerts` パネルの使い分けを共有し、レビュー手順を簡素化
- 問題や既知の制約に遭遇した場合は、案内されているフィードバック discussion に集約して報告

## Hierarchy view in GitHub Projects is now generally available

**URL:** https://github.blog/changelog/2026-03-19-hierarchy-view-in-github-projects-is-now-generally-available

### 要約
GitHub Projects の hierarchy view が一般提供になり、新しい project view では既定で有効になります。既存 view でも `Show hierarchy` を有効化でき、加えて sub-issue filter の見つけやすさ向上、自動 filter 適用、アクセシビリティ改善も提供されます。あわせて、issue template での `@copilot` 自動割り当てと、maintainer 向け blank issue の扱いも改善されました。

### ユーザーにとって良いこと
親子関係を持つ issue を Projects 上でより自然に扱えるようになり、計画管理と進捗把握がしやすくなります。issue template と blank issue の改善により、定型タスクの自動化と maintainer の運用自由度も上がります。

### アクションプラン
- 既存の Projects view では `View` メニューから `Show hierarchy` を有効化し、親子 issue 管理に適したビューを標準化
- issue template で Copilot に定型タスクを委譲したい場合は、`assignees` に `@copilot` を追加
- `blank_issues_enabled: false` を運用しているリポジトリでは、maintainer が blank issue を作成できる前提でガイドラインを更新

### リファレンス
- [Hierarchy view in GitHub Projects is now generally available](https://github.blog/changelog/2026-03-19-hierarchy-view-in-github-projects-is-now-generally-available)
