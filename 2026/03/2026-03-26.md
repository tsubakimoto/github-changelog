## Gemini 3 Pro deprecated

**URL:** https://github.blog/changelog/2026-03-26-gemini-3-pro-deprecated

### 要約
GitHub Copilot の各体験で Gemini 3 Pro が非推奨になり、代替として Gemini 3.1 Pro が案内されています。Copilot Chat、inline edits、ask/agent モード、コード補完を含む広い範囲に影響します。

### ユーザーにとって良いこと
利用中のモデルが明確になり、サポート継続中のモデルへ移行しやすくなります。Enterprise 管理者はモデルポリシーを通じて利用可能な代替モデルを制御できます。

### アクションプラン
- Copilot のモデル選択で Gemini 3.1 Pro が利用できるか確認してください。
- Copilot Enterprise 管理者は、モデルポリシーで代替モデルの有効化状況を確認してください。
- ⚠️ Gemini 3 Pro を前提にしたワークフローや統合は、サポート対象モデルへ切り替えてください。

### リファレンス
- [Gemini 3 Pro deprecated](https://github.blog/changelog/2026-03-26-gemini-3-pro-deprecated)

## Credential revocation API now supports GitHub OAuth and GitHub app credentials

**URL:** https://github.blog/changelog/2026-03-26-credential-revocation-api-now-supports-github-oauth-and-github-app-credentials

### 要約
Credential revocation API が拡張され、OAuth トークンや GitHub App のユーザー向け・更新用トークンも一括失効できるようになりました。公開された資格情報を見つけたときに、リポジトリ所有者でなくても被害を広げずに対処できます。

### ユーザーにとって良いこと
漏えいした認証情報の無効化を素早く自動化でき、インシデント対応の初動を短縮できます。対象トークンが増えたことで、実運用の検知・封じ込めフローをまとめやすくなります。

### アクションプラン
- 漏えい検知ツールや IR 手順に、Credential revocation API の呼び出しを追加してください。
- 対応対象のトークン種別が増えたため、PAT 以外の GitHub OAuth / GitHub App トークンも失効対象に含めてください。
- ⚠️ この API は unauthenticated で利用されるため、60 req/hour と 1,000 tokens/request の制限を踏まえてバッチ設計してください。

### リファレンス
- [Credential revocation API now supports GitHub OAuth and GitHub app credentials](https://github.blog/changelog/2026-03-26-credential-revocation-api-now-supports-github-oauth-and-github-app-credentials)

## Ask @copilot to resolve merge conflicts on pull requests

**URL:** https://github.blog/changelog/2026-03-26-ask-copilot-to-resolve-merge-conflicts-on-pull-requests

### 要約
PR のコメントで `@copilot` に依頼すると、Copilot coding agent が merge conflict を解消できるようになりました。エージェントはクラウド環境で変更、テスト、push まで実行します。

### ユーザーにとって良いこと
競合解消のためにローカルへ切り替える手間が減り、PR 上の会話の流れを保ったまま作業できます。CI 修正やレビュー対応と同じ流れで conflict 解消も依頼でき、PR の停滞を減らせます。

### アクションプラン
- PR の衝突解消フローに `@copilot Merge in main and resolve the conflicts` を追加してください。
- Copilot Business / Enterprise を使っている場合は、管理者が coding agent を有効化しているか確認してください。
- ⚠️ 既存の手動マージ手順やブランチ保護ルールと矛盾しないか検証してください。

### リファレンス
- [Ask @copilot to resolve merge conflicts on pull requests](https://github.blog/changelog/2026-03-26-ask-copilot-to-resolve-merge-conflicts-on-pull-requests)

## New pull requests dashboard is in public preview

**URL:** https://github.blog/changelog/2026-03-26-new-pull-requests-dashboard-is-in-public-preview

### 要約
`github.com/pulls` に刷新された pull requests ダッシュボードが public preview で追加されました。Inbox と Saved views により、対応が必要な PR を整理し、検索条件を保存して再利用できます。

### ユーザーにとって良いこと
大量の PR を、レビュー待ち・修正待ち・マージ待ちで素早く分けて追いやすくなります。保存済みビューでチームごとの見方を再現しやすく、日々の triage が効率化されます。

### アクションプラン
- `github.com/pulls` を開いて Inbox と Saved views を試してください。
- レビュー担当者は、よく使う検索条件を Saved view として保存してください。
- ⚠️ Public preview なので、重要な運用に組み込む前に UI/検索結果の挙動を確認してください。

### リファレンス
- [New pull requests dashboard is in public preview](https://github.blog/changelog/2026-03-26-new-pull-requests-dashboard-is-in-public-preview)

## View Agentic Workflow configs in the Actions run summary

**URL:** https://github.blog/changelog/2026-03-26-view-agentic-workflow-configs-in-the-actions-run-summary

### 要約
Agentic Workflow の markdown 設定を、GitHub Actions の run summary から直接確認できるようになりました。実行時に使われた設定をその場で追えるため、レビューや診断がしやすくなります。

### ユーザーにとって良いこと
ワークフロー定義と実行結果を行き来する手間が減り、再現性の確認が簡単になります。問題のある run を見たときに、実際の設定を素早く照合できます。

### アクションプラン
- Agentic Workflows を使っているリポジトリで、run summary に設定が表示されることを確認してください。
- 失敗 run の調査手順に、run summary の設定確認を追加してください。
- ⚠️ 既存の診断手順が別ページ前提なら、run summary ベースに更新してください。

### リファレンス
- [View Agentic Workflow configs in the Actions run summary](https://github.blog/changelog/2026-03-26-view-agentic-workflow-configs-in-the-actions-run-summary)

## Agent activity in GitHub Issues and Projects

**URL:** https://github.blog/changelog/2026-03-26-agent-activity-in-github-issues-and-projects

### 要約
GitHub Issues と Projects で、coding agent のセッション状況が直接見えるようになりました。Issue の assignee の下や project の table/board に、queued・working・waiting for review・completed といった状態が表示されます。

### ユーザーにとって良いこと
作業の進み具合を issue や project の画面から一目で把握でき、エージェント作業を計画に組み込みやすくなります。大きな作業群でも、どれが進行中か追跡しやすくなります。

### アクションプラン
- Issues で coding agent を使っているリポジトリは、assignee 配下の session 表示を確認してください。
- Projects の table/board で `Show agent sessions` を有効にして可視化してください。
- **自動適用**: GitHub Issues / Projects で coding agent が使えるリポジトリに表示されます。

## Custom images for GitHub-hosted runners are now generally available

**URL:** https://github.blog/changelog/2026-03-26-custom-images-for-github-hosted-runners-are-now-generally-available

### 要約
GitHub-hosted runners の custom images が一般提供になりました。GitHub のベースイメージを起点に、必要なツールや設定を組み込んだ独自イメージを作成でき、ビルド環境を標準化しやすくなります。

### ユーザーにとって良いこと
セットアップ時間を短縮し、ワークフローの再現性とセキュリティを高めやすくなります。既に public preview を使っていた場合は継続利用できます。

### アクションプラン
- custom images を使う場合は、ベースイメージ上に必要な依存関係・証明書・設定を組み込んでください。
- public preview から使っていたリポジトリは、既存イメージがそのまま動くことを確認してください。
- **自動適用**: 既存の custom images と workflows は引き続き動作します。新規利用時のみ設計・導入が必要です。

### リファレンス
- [Custom images for GitHub-hosted runners are now generally available](https://github.blog/changelog/2026-03-26-custom-images-for-github-hosted-runners-are-now-generally-available)
