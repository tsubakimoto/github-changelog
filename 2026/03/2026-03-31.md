## GitHub secret scanning — coverage update

**URL:** https://github.blog/changelog/2026-03-31-github-secret-scanning-nine-new-types-and-more

### 要約
GitHub secret scanning に 9 種類の新しい検出器が追加され、Langchain、Salesforce、Figma などのプロバイダーが対象になりました。あわせて、Figma、Google、OpenVSX、PostHog の一部シークレットは push protection の既定対象になり、npm シークレットには有効性チェックも追加されています。さらに、push protection のパターン名から関連アラート一覧へ直接移動できるようになりました。

### ユーザーにとって良いこと
新しい種類のシークレット漏えいをより早く検出でき、push 時点でのブロックも強化されます。担当者は「検出したが無効かもしれない」ではなく、有効性確認済みのアラートを優先して対応しやすくなります。

### アクションプラン
- secret scanning を有効にしているリポジトリで、新規検出対象のシークレットがないか確認してください。
- Figma、Google、OpenVSX、PostHog の push protection 既定対象が自組織の運用に与える影響を確認してください。
- npm の `npm_access_token` アラートは有効性チェック結果を見て、優先度を付けて対応してください。
- ⚠️ push protection の挙動が増えるため、CI や開発者のコミット運用でブロックが増えないか事前に試してください。

## CodeQL 2.25.0 adds Swift 6.2.4 support

**URL:** https://github.blog/changelog/2026-03-31-codeql-2-25-0-adds-swift-6-2-4-support

### 要約
CodeQL 2.25.0 が公開され、Swift 6.2.4 の解析に対応しました。Java/Kotlin の制御フローグラフは全面的に書き換えられ、C# 14 の partial constructors や WebSocket 由来の taint source、ブラウザ向け source kind のサポートも追加されています。2.25.1 も利用可能ですが、こちらは軽微な不具合修正のみです。

### ユーザーにとって良いこと
より新しい言語機能に対応したまま、code scanning の精度向上を受けられます。特に Java/Kotlin の解析精度改善は、誤検出や見逃しの削減につながる可能性があります。

### アクションプラン
- GitHub code scanning を使っている場合、現在の CodeQL バージョンを 2.25.0 以降に更新してください。
- Swift、Java/Kotlin、C#、JavaScript/TypeScript を使うリポジトリでは、解析結果の差分を確認してください。
- GHES を利用している場合は、将来の GHES 取り込み予定を確認し、必要なら手動アップグレードを検討してください。
- ⚠️ Java/Kotlin の CFG 変更により、既存のアラート件数や内容が変わる可能性があります。

## CodeQL pull requests insights on security overview now cover all protected branches

**URL:** https://github.blog/changelog/2026-03-31-codeql-pull-requests-insights-on-security-overview-now-cover-all-protected-branches

### 要約
Security overview の CodeQL pull request insights が、デフォルトブランチだけでなく全ての protected branches を集計対象にするようになりました。9 つの insight タイルと CSV ダウンロードの数値が、より実態に近い値になります。過去データもさかのぼって変わるため、履歴値が増える可能性があります。

### ユーザーにとって良いこと
組織全体での Autofix の効果や alert 解消状況を、より正確に把握できます。ブランチごとの差を気にせず、保護されたブランチ全体での改善状況を一目で追えるようになります。

### アクションプラン
- Security overview の CodeQL insights を見直し、保護ブランチ全体の数値として解釈してください。
- CSV エクスポートやダッシュボードの値が過去より増えても、仕様変更による再集計だとチームへ周知してください。
- Autofix の導入効果を報告している場合は、集計範囲の変更を前提にレポートを更新してください。
- ⚠️ 過去データがさかのぼって変わるため、以前の比較レポートは再計算が必要になる場合があります。

## EU data residency region expanding to include EFTA countries

**URL:** https://github.blog/changelog/2026-03-31-eu-data-residency-region-expanding-to-include-efta-countries

### 要約
GitHub Enterprise Cloud on ghe.com の EU data residency region が、2026-05-01 から EFTA 諸国の Azure インフラにも拡大されます。対象はノルウェーとスイスで、既存の EU 加盟国リージョンに加わります。セキュリティ統制、暗号化、認証要件は変わらず、通常は顧客対応不要です。

### ユーザーにとって良いこと
EU データレジデンシーの選択肢が広がり、より柔軟なリージョン運用が可能になります。EU Boundary に沿った地理的な整合性が高まり、コンプライアンス面の説明もしやすくなります。

### アクションプラン
- EU-region の Enterprise を運用している場合、5 月 1 日以降にノルウェー/スイスが対象に含まれることを把握してください。
- 「EU 加盟国のみに厳格に限定したい」要件がある組織は、5 月 1 日までに GitHub アカウントチームまたは Support に連絡してください。
- コンプライアンス資料や社内説明文書に、EFTA を含む新しい範囲を反映してください。
- ⚠️ EU 加盟国限定の要件がある場合は、この変更をそのまま受け入れず、事前確認が必要です。
