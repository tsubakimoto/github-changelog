## Copilot coding agent works faster with semantic code search

**URL:** https://github.blog/changelog/2026-03-17-copilot-coding-agent-works-faster-with-semantic-code-search

### 要約
Copilot coding agent が semantic code search を利用できるようになり、単なる文字列一致ではなく意味ベースで関連コードを探せるようになりました。GitHub の検証では、品質を落とさずにタスク完了時間が約 2% 短縮されています。

### ユーザーにとって良いこと
コードの正確な名前や配置を知らなくても、Copilot が必要な箇所を見つけやすくなります。大きなコードベースや曖昧な依頼で、エージェントの探索速度と精度が向上します。

### アクションプラン
- 設定変更は不要なので、普段どおり Copilot coding agent を使って体感差を確認
- 大規模リポジトリや曖昧な変更依頼で、semantic search の効果が高いか観察
- 自動適用の改善として、事前の検索指示を減らして agent の挙動を評価

## Dependabot now detects malware in npm dependencies

**URL:** https://github.blog/changelog/2026-03-17-dependabot-now-detects-malware-in-npm-dependencies

### 要約
Dependabot が npm 依存関係に含まれる既知の悪性バージョンを検出できるようになりました。malware alerts は従来の CVE ベース alerts とは分離され、opt-in で有効化し、ルールで細かく調整できます。

### ユーザーにとって良いこと
サプライチェーン攻撃や悪意あるパッケージ混入への可視性が高まります。CVE と malware を分けてトリアージできるため、ノイズを抑えながらリスクに対応しやすくなります。

### アクションプラン
- repository / organization / enterprise の Dependabot 設定で **Malware alerts** を有効化
- private package と同名の public package がある場合は、rule 設定で誤検知を抑制
- ⚠️ opt-in 機能かつ現時点では npm 対象なので、必要な範囲を明示して段階導入

### リファレンス
- [Dependabot now detects malware in npm dependencies](https://github.blog/changelog/2026-03-17-dependabot-now-detects-malware-in-npm-dependencies)

## Push protection exemptions for roles, teams, and apps

**URL:** https://github.blog/changelog/2026-03-17-push-protection-exemptions-for-apps-teams-and-roles

### 要約
secret scanning push protection を有効化している organizations で、特定の roles、teams、apps を exempt actor として指定できるようになりました。該当 actor の push では push protection がスキップされ、bypass request も生成されません。

### ユーザーにとって良いこと
信頼済みの自動化や特定チームに対して、push protection の摩擦を抑えた運用ができます。例外管理を組織レベルで明示的に行えるため、統制と柔軟性の両立に役立ちます。

### アクションプラン
- enterprise / organization の security configurations で、本当に exempt が必要な roles / teams / apps だけを登録
- 例外付与した actor を定期監査し、不要になった exempt は削除
- ⚠️ push protection を完全に迂回する設定なので、最小権限の原則で運用し、広範囲な exempt を避ける

### リファレンス
- [Push protection exemptions for roles, teams, and apps](https://github.blog/changelog/2026-03-17-push-protection-exemptions-for-apps-teams-and-roles)

## Code Quality permissions removed from security manager role

**URL:** https://github.blog/changelog/2026-03-17-code-quality-permissions-removed-from-security-manager-role

### 要約
security manager role では、repository administrator でない限り GitHub Code Quality の有効化・無効化ができなくなりました。権限は least privilege の原則に沿って整理され、Code Quality の管理主体が repository admins に限定されます。

### ユーザーにとって良いこと
権限境界がより明確になり、セキュリティ関連ロールに不要な操作権限が残りにくくなります。権限分離を重視する組織では、運用監査もしやすくなります。

### アクションプラン
- Code Quality を security manager role で操作していた運用がないか確認
- repository admins 側へ作業移管し、権限依存の runbook を更新
- ⚠️ 既存運用に影響する権限変更なので、管理者フローと問い合わせ窓口を明確にする

### リファレンス
- [Code Quality permissions removed from security manager role](https://github.blog/changelog/2026-03-17-code-quality-permissions-removed-from-security-manager-role)

## GitHub Code Quality: Batch apply quality suggestions on pull requests

**URL:** https://github.blog/changelog/2026-03-17-github-code-quality-batch-apply-quality-suggestions-on-pull-requests

### 要約
Pull request の **Files changed** タブで、GitHub Code Quality の修正提案を batch でまとめて適用できるようになりました。複数の指摘に対して一括で修正し、スキャンも 1 回でまとめて回しやすくなります。

### ユーザーにとって良いこと
指摘ごとに個別 remediation と再解析を繰り返す必要が減り、PR の修正とレビューが速くなります。Code Quality を使った改善サイクルの負荷を下げる実用的な UI 改善です。

### アクションプラン
- PR 作成者とレビュアーは **Files changed** で batch apply を試し、複数指摘の remediation 手順を簡素化
- 一括修正後に single scan を回す前提で、レビュー手順や説明テンプレートを更新
- github.com では GA 済みのため、まずは通常運用で使い勝手を確認

### リファレンス
- [GitHub Code Quality: Batch apply quality suggestions on pull requests](https://github.blog/changelog/2026-03-17-github-code-quality-batch-apply-quality-suggestions-on-pull-requests)
